# 策略回测报告目录按策略名分层 (XTR-012)

## Intent
当前策略回测报告目录长期使用任务号（如 `XTR-011`）作为集合根目录，容易把任务管理信息混入程序运行结构，也不利于未来多策略并行研发与报告管理。  
本任务目标是将回测报告目录改为“按策略名分层”，让目录结构直接反映策略维度，而不是任务维度。

## Requirement
- 功能目标
  - 提供统一的策略回测报告目录构造逻辑：`reports/backtests/strategy/<strategy_slug>/<run_id>/`。
  - `run_id` 由时间戳与策略名组成，避免不同策略同一时间运行时目录冲突。
  - `report_hub.html` / `report_hub_index.json` 统一生成在：`reports/backtests/strategy/`。
  - 报告服务输出的 hub 链接不再绑定任务号，按当前目录自动发现策略集合 hub。
  - 每次回测报告写出时必须固定生成输入快照（无开关）：
    - `snapshots/price_input.parquet`
    - `snapshots/action_input.parquet`
    - `snapshots/snapshot_manifest.json`
  - 快照 manifest 需记录行数、时间范围与文件哈希，确保即使元数据变更也可复现实验输入。
- 非目标 / 范围外
  - 不迁移历史目录结构（旧目录继续可访问）。
  - 不修改回测指标计算逻辑与交易撮合逻辑。
  - 不引入数据库，仅保持本地文件结构改造。
- 输入输出或接口
  - 新增“按策略名生成 report_root”的公开方法，供策略回测调用。
  - 保留现有 `write_event_driven_outputs(report_root=...)` 兼容能力，避免破坏已有调用方。

## Design
- 核心思路与架构
  - 在 `event_driven` 模块新增策略目录辅助函数（策略名 slug + UTC 时间戳 run_id）。
  - 新增策略语义的输出包装方法，内部调用现有 `write_event_driven_outputs`。
  - 报告服务 `report_server.py` 在 `start/restart` 前重建 `report_hub_index.json`，按文件系统实际目录生成集合索引。
- 数据/接口/模型
  - 新目录：
    - `reports/backtests/strategy/<strategy_slug>/<run_id>/summary.json`
    - `reports/backtests/strategy/<strategy_slug>/<run_id>/event_driven_report.html`
    - `reports/backtests/strategy/report_hub_index.json`
  - `run_id` 规范：`YYYYMMDDTHHMMSSZ_<strategy_slug>`。
- 风险与权衡
  - 风险：调用方未切换到新入口时，仍可能使用旧目录。
  - 权衡：保持旧接口兼容，同时提供新入口与文档示例，渐进切换。

## Acceptance
- 新增策略目录辅助函数单测通过，能稳定输出 `reports/backtests/strategy/<strategy_slug>/<run_id>`。
- 使用策略语义输出接口写报告时，产物落盘到策略目录，且 `report_hub_index.json` 位于 `reports/backtests/strategy/`。
- `report_server.py status/start` 输出不再硬编码任务号路径，能显示动态发现的 hub。
- 现有 `tests/unit/backtests/test_event_driven.py` 全量通过。
- 报告产物中始终存在 `snapshots/*` 三个文件，且 `snapshot_manifest.json` 的行数与哈希与快照文件一致。

## Notes
- 本任务仅涉及策略回测报告目录，不影响数据下载/重采样/对账模块。
