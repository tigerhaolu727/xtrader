# K线下载目录结构改造与兼容性评估 (XTR-005)

## Intent
当前 K 线本地存储目录采用 `key=value` 分区风格（如 `exchange=bitget/.../month=MM/part.parquet`），对人工定位不够直观。需求是将目录改为“仅值路径”并将月份从目录改为文件名，便于浏览与管理：
- 旧：`data/klines/exchange=bitget/market=linear_swap/symbol=BTCUSDT/interval=5m/year=YYYY/month=MM/part.parquet`
- 新：`data/klines/bitget/linear_swap/BTCUSDT/5m/YYYY/MM.parquet`
同时评估此改动对既有数据和逻辑的影响与风险。

## Requirement
- 功能目标
  - 修改 `KlineLocalStore` 的数据集路径规则，去掉 `exchange= / market= / symbol= / interval= / year=` 前缀，仅保留值目录。
  - 修改月分区存储规则：
    - 目录从 `.../year=YYYY/month=MM/part.parquet` 改为 `.../YYYY/MM.parquet`。
  - 读取、写入、删除、扫描分区（`_partition_file/_partition_files/delete_dataset`）全部兼容新结构。
  - 保持索引文件位置不变（`data/klines_index/{exchange}.{market}.{symbol}.{interval}.json`），避免额外迁移复杂度。
- 非目标 / 范围外
  - 本任务不自动迁移旧目录历史数据（仅说明风险与人工迁移策略）。
  - 不修改 K 线记录 schema 与下载分页逻辑。
  - 不变更 CLI 参数格式。
- 输入输出或接口
  - 输入：与现有 `download_klines` CLI 保持一致。
  - 输出：新目录结构下的 Parquet 文件与原有索引文件。

## Design
- 核心思路与架构
  - 仅改存储层 `KlineLocalStore` 的路径构造与扫描匹配，不改上层 `KlineSyncEngine` 逻辑。
  - 将路径变化限定在以下方法：
    - `dataset_path`
    - `_partition_file`
    - `_partition_files`
    - `delete_dataset`（依赖 `dataset_path`）
  - 通过单元测试覆盖“写入后索引正常、缺口检测正常、路径命中正常”。
- 数据/接口/模型
  - 新数据路径：
    - `data/klines/{exchange}/{market_type}/{symbol}/{interval}/{YYYY}/{MM}.parquet`
  - 索引路径保持：
    - `data/klines_index/{exchange}.{market_type}.{symbol}.{interval}.json`
- 风险与权衡
  - 风险 1：旧路径数据不会被新 glob 命中，表现为“看起来没有历史覆盖”并触发重复下载。
    - 对策：在变更说明中明确“需手动迁移旧目录或接受重新下载”。
  - 风险 2：月份目录改为文件后，未来若要同月多文件滚动（part-0001）扩展性降低。
    - 对策：首期采用单文件 `MM.parquet`，后续可升级为 `MM.part-xxxx.parquet`。
  - 风险 3：隐藏文件（`.DS_Store`）噪音仍可能存在，但不影响 `*.parquet` 匹配。
  - 权衡：新结构可读性更好，路径更短；代价是与旧数据布局不兼容，需要一次性迁移决策。

## Acceptance
- `download_klines` 下载后文件路径满足：`data/klines/bitget/linear_swap/BTCUSDT/5m/2024/01.parquet` 形式。
- `KlineLocalStore` 的 `_partition_files` 仅匹配新结构文件并能正常读取。
- `get_coverage` / `find_missing_ranges` / `upsert_records` 在新结构下行为正确，单元测试通过。
- 旧路径数据不做自动兼容时，应在文档中明确影响与迁移建议。

## Notes
- 建议在 README 或变更说明新增“目录结构升级”小节。
- 可选迁移脚本（后续任务）：扫描旧路径并重命名到新路径，迁移后重建索引。
