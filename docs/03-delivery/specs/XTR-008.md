# FeatureStore 因子产物存储与复现管理 (XTR-008)

## Intent
当前项目已完成统一因子抽象（XTR-007）与重采样/防泄漏/对账底座（XTR-006），但缺少统一因子产物仓库，导致因子计算结果在评估、回测、策略模块之间难以复用与追溯。本任务目标是建设 `FeatureStore`，实现因子产物标准化存储、版本管理、复现回放与质量治理，支撑后续因子评估与策略研发闭环。

## Requirement
- 功能目标
  - 提供统一因子产物存储能力，标准化保存 `FactorResult`。
  - 支持因子身份管理：`factor_id + factor_version + params_hash + data_version` 唯一标识。
  - 支持幂等写入与重复请求缓存命中，避免重复计算污染。
  - 支持按 `factor_id/version/params_hash/symbol/time_range` 查询读取。
  - 支持 identity 索引清单（`identities.jsonl`）维护，记录产物身份、覆盖范围、访问状态。
  - 提供 CLI 入口，支持 identity 列表查询与 cleanup 策略执行（dry-run/execute）。
  - 支持运行元数据追溯：`run_id/task_id/code_version/input_versions/created_at`。
  - 支持产物质量指标记录：覆盖率、缺失率、质量标记。
  - 支持评估与回测模块复用同一份因子产物数据。
  - 支持清理策略：`dry-run` 预演、`mark cold` 冷数据标记、`delete` 执行删除与审计报告输出。
- 非目标 / 范围外
  - 本任务不实现分布式 Feature 服务与在线低延迟 serving。
  - 本任务不引入外部数据库（延续本地文件系统分层方案）。
  - 本任务不实现完整权限系统与多租户隔离。
- 输入输出或接口
  - 输入：
    - 标准因子结果 `FactorResult`
    - 因子身份信息（`factor_id/version/params_hash/data_version`）
    - 运行元数据（`run_id/task_id/code_version/input_versions`）
  - 输出：
    - 持久化因子产物分区数据（Parquet）
    - 元数据索引（JSON）
    - 质量报告与写入摘要

## Design
- 核心思路与架构
  - 采用 `FeatureStore` 作为因子产物唯一读写入口，解耦 `FactorEngine` 与 `Evaluator/Backtest`。
  - 分层职责：
    - `FeatureStoreWriter`：身份校验、幂等写入、分区落盘。
    - `FeatureStoreReader`：条件查询、窗口读取、版本筛选。
    - `FeatureMetadataStore`：运行元数据与质量指标记录。
  - 写入流程：校验 schema -> 计算 identity -> 幂等检查 -> 写入数据 -> 更新元数据。
  - 读取流程：按 identity + 过滤条件加载 -> 返回标准 DataFrame 给评估/回测模块。
- 数据/接口/模型
  - 产物最小字段：
    - `timestamp`, `symbol`, `factor_id`, `factor_version`, `value`
  - 扩展字段（可选）：
    - `is_valid`, `quality_flag`, `missing_ratio`, `extra`
  - 存储路径建议：
    - `data/features/{exchange}/{market}/{symbol}/{interval}/factor_id={id}/version={v}/params_hash={h}/YYYY/MM.parquet`
  - 元数据路径建议：
    - `data/features_meta/{factor_id}/{version}/{params_hash}/{run_id}.json`
  - 报告路径建议：
    - `reports/features/XTR-008/{run_id}/feature_quality.md`
    - `reports/features/XTR-008/cleanup_{run_id}/cleanup_report.md`
  - identity 索引建议：
    - `data/features_index/identities.jsonl`
    - 字段：`exchange/market_type/symbol_scope/interval/factor_id/factor_version/params_hash/data_version/latest_run_id/row_count/min_timestamp/max_timestamp/created_at/updated_at/last_accessed_at/pinned/status`
  - 对外接口建议：
    - `put(result, identity, metadata) -> write_summary`
    - `get(identity, start_time, end_time, symbols=None) -> DataFrame`
    - `exists(identity) -> bool`
    - `get_metadata(run_id or identity) -> dict`
    - `list_identities(factor_id=None, status=None) -> list[dict]`
    - `cleanup_identities(dry_run=True, keep_recent_params=20, cold_after_days=90, delete_after_days=180) -> cleanup_summary`
- 风险与权衡
  - 风险：身份定义不完整导致“同因子不同输入版本”覆盖写入。
    - 对策：强制 identity 包含 `data_version`，并写入元数据索引。
  - 风险：幂等策略不当导致重复数据或误判命中。
    - 对策：统一主键与覆盖策略，写入前后做记录数校验。
  - 风险：下游读取口径不统一导致评估结果不可比。
    - 对策：严格输出 schema 与统一读取 API，禁止绕过 `FeatureStore` 直读散落文件。

## Acceptance
- 至少 2 个示例因子产物可通过 `FeatureStore.put` 写入并通过 `FeatureStore.get` 查询返回。
- 同 identity 重复写入不产生重复记录，幂等行为可验证。
- 产物可通过 `run_id` 追溯到输入版本与代码版本。
- 评估模块可直接消费 `FeatureStore` 数据计算至少 1 个指标（例如 IC）。
- 回测模块可直接读取 `FeatureStore` 产物完成端到端样例回测。
- identity 索引清单生成并可查询，`last_accessed_at/status` 能随读写更新。
- 清理策略支持 dry-run 与执行模式，删除动作可追溯并产出 cleanup 报告。
- CLI 支持 `--list-identities` 与 cleanup 参数化执行，输出结构化 JSON 结果。
- 守门检查与相关单测通过，结果可复现。
