# 回测与报告解耦及离线报告UI规范 (XTR-013)

## Intent
当前事件驱动回测模块同时承担“回测计算”和“HTML 报告生成”职责，导致模块边界不清、维护成本高、UI 调整易影响回测稳定性。  
本任务目标是将两者完全解耦：回测程序只输出标准数据产物；报告展示由独立离线 UI 读取标准产物进行可视化。

## Requirement
- 功能目标
  - 回测程序中移除旧报告生成功能，不再生成 `event_driven_report.html`、`event_driven_report.md`、`report_hub.html`、`report_hub_index.json`。
  - 回测程序仅输出标准化 run 产物，目录规范为 `reports/backtests/strategy/<strategy_slug>/<run_id>/`。
  - `run_manifest.json` 作为唯一入口索引，记录 schema 版本、artifact 路径、行数、sha256、时间范围。
  - 报告 UI 支持离线运行（无需本地服务），用户只需选择一次 `run_manifest.json` 或 run 目录，系统自动加载对应数据文件。
  - UI 必须展示：K 线、交易记录（Trade Ledger）、资金曲线（Equity Curve）、Signal vs Execution Timeline。
  - K 线、资金曲线、Timeline 使用统一 UTC X 轴；K 线 Y 轴随当前可视窗口动态缩放，支持缩放与拖动。
  - 交易记录支持选中联动：定位并高亮 K 线与时间线对应区间。
  - 基础行情最小周期固定为 5m；策略若使用 15m/1H/1D 等更大周期，需由 5m 数据重采样聚合产生，并将本次 run 实际使用的重采样结果落盘。
  - UI 必须支持长历史回测数据可视化（典型规模：5m 三年 > 30 万根 K 线、交易上万条、Timeline/Equity 数万至数十万点），不得采用“全量一次加载 + 全量一次绘制”模式。
  - UI 采用“视窗优先（viewport-first）”读取策略：默认仅加载最近窗口（如 7D 或 30D），按时间窗增量加载。
  - K 线渲染必须采用像素预算控制：同屏可视 K 线数量上限由图宽驱动（例如不超过约 `2 * chart_width_px`），超限时自动切换更粗粒度或桶聚合 OHLC。
  - Timeline 在大范围下需支持聚合展示（密度/计数），放大后再展示单事件连线。
  - Trade Ledger 必须支持虚拟滚动或分页，避免一次渲染上万行 DOM。
  - UI 数据解析与聚合需支持后台线程（Web Worker）执行，避免主线程阻塞。
- 非目标 / 范围外
  - 不迁移任何旧历史报告文件。
  - 不提供旧格式到新格式的转换脚本。
  - 本阶段不实现复杂后端服务能力（如多用户、权限、数据库索引）。
- 输入输出或接口
  - 回测输出（建议最低集）：
    - `run_manifest.json`
    - `summary.json`
    - `diagnostics.json`
    - `snapshots/base/price_5m.parquet`
    - `snapshots/action_input.parquet`
    - `snapshots/resampled/resampled_manifest.json`
    - `snapshots/resampled/price_<tf>.parquet`（仅落盘本次策略实际使用的更大周期，如 `price_15m.parquet`、`price_1h.parquet`、`price_1d.parquet`）
    - `timelines/signal_execution.parquet`
    - `ledgers/trades.parquet`（可兼容导出 csv）
    - `curves/equity_curve.parquet`（可兼容导出 csv）
    - `*/chunks/` + 对应 chunk manifest（用于按时间窗增量加载；适用于 prices/trades/equity/timeline）
  - 时间字段统一 UTC，核心字段名统一：`signal_time`、`execution_time`、`entry_time`、`exit_time`。
  - `snapshots/resampled/resampled_manifest.json` 需记录重采样来源与规则（例如 open/ high/ low/ close 聚合口径、label/closed、timezone）、行数、时间范围、sha256、生成器版本。

## Design
- 核心思路与架构
  - 回测层（Backtest Core）仅负责计算与落盘；不再承担报告渲染。
  - 展示层（Report UI）独立读取 run 标准产物，负责交互与可视化。
  - 以 `run_manifest.json` 驱动数据加载，避免手动多文件选择造成错配。
  - 展示层采用“三层读取模型”：
    - 元数据层：`run_manifest.json`（统一索引与校验）
    - 数据块层：按时间块读取 prices/trades/equity/timeline
    - 视图层：按当前视窗进行二次聚合后渲染
- 数据/接口/模型
  - 定义统一 `schema_version`（例如 `bt_run_v1`），确保版本演进可控。
  - manifest 中每个 artifact 需有 `path/rows/sha256/min_time/max_time`（按文件类型可选时间范围字段）。
  - `run_manifest.json` 需显式标注基础周期（`base_timeframe=5m`）以及本次 run 使用的重采样周期列表。
  - 重采样数据采用“双层快照”策略：
    - 基础真值层：`snapshots/base/price_5m.parquet`
    - 派生聚合层：`snapshots/resampled/price_<tf>.parquet` + `resampled_manifest.json`
  - UI 内部使用统一时间窗状态，同步驱动 K 线、资金曲线、时间线三图。
  - K 线多分辨率建议最低支持：`5m/15m/1h/4h/1d`，并支持窗口内桶聚合（OHLC）兜底。
  - Equity 建议支持降采样（LTTB 或 min-max bucket），默认渲染点数限制在可视像素预算内。
  - Timeline 建议支持双模式：
    - 聚合模式：大时间窗显示事件密度/计数
    - 明细模式：小时间窗显示 signal->execution 连线与 hover 明细
- 风险与权衡
  - 风险：一次性移除旧报告可能影响当前依赖旧页面的使用习惯。
  - 权衡：明确“不迁移旧历史报告”，换取架构简化与后续迭代速度。
  - 风险：纯离线模式下大数据加载压力较高。
  - 权衡：本阶段通过分块读取、降采样、局部渲染、动态 Y 轴优化保证可用性，后续再评估服务化。
  - 风险：多图联动与异步加载并存时，易出现时间窗错位。
  - 权衡：采用“单一时间窗状态源 + 版本号防抖”策略，确保 K 线/Equity/Timeline 同步一致。

## Acceptance
- 可验证的验收标准（列举具体检查或指标）
  - 新回测 run 不再产出旧报告文件（`event_driven_report.*`、`report_hub.*`）。
  - 新回测 run 必须产出标准化数据文件与 `run_manifest.json`，且 snapshot 固定落盘（无开关）。
  - UI 通过单一 manifest 入口可自动加载并展示四类核心信息（K 线/交易/资金/Timeline）。
  - 对于使用 15m/1H/1D 等策略 run，目录中存在对应重采样快照与 `resampled_manifest.json`，且可追溯到 `price_5m.parquet`。
  - 三图 X 轴统一 UTC 时间范围，缩放拖动后时间窗同步。
  - K 线 Y 轴在不同缩放区间下动态变化，视觉高度可读、无固定轴压扁问题。
  - 旧历史报告不迁移、不转换，移除策略与流程在文档中明确。
  - 在“5m 三年（>30 万 K线）+ 上万 trades + 数万 timeline/equity 点”样例下：
    - 默认窗口首屏可完成加载与渲染（无浏览器卡死）。
    - 缩放/拖动时三图联动可持续操作，交互无明显阻塞。
    - Trade Ledger 可浏览且不因一次性 DOM 渲染导致页面失去响应。

## Notes
- 依赖、文档链接、额外限制
  - 本 Spec 仅定义解耦与展示规范，不包含具体前端视觉风格稿。
  - 后续开发需先按本任务补充验证证据，再进入编码与回归测试。
  - 推荐在验证中提供固定“压力样例 run”（例如 BTCUSDT 5m 三年）以保证性能回归可比较。
