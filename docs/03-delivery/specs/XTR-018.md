# Feature Engine 指标实现 v1（核心连续值） (XTR-018)

## Intent
当前策略文档（`trading_strategy.md`）需要一组稳定、可复用的技术指标基础能力，供后续五维评分与信号层使用。  
本任务目标是定义并实现 `feature_engine` 的指标层 v1：指标层仅负责“核心连续值计算”，不承担金叉/死叉等事件判定，确保模块边界清晰、可测试、可扩展。

## Requirement
- 功能目标
  - 在 `src/xtrader/strategies/feature_engine/indicators/` 下实现指标族计算能力，支持参数化实例（同一指标族不同参数）。
  - 为所有指标族提供统一抽象接口（`BaseIndicator`），保障调用方式一致。
  - 指标族按“一个指标族一个文件”组织，并按分类目录管理：`trend/oscillator/volatility/volume`。
  - v1 覆盖以下指标族：
    - 趋势：`MA`、`EMA`、`MACD`、`DMI`
    - 振荡：`RSI`、`KD`、`WR`
    - 波动率：`ATR`、`Bollinger Bands`、`StdDev`
    - 成交量：`Volume MA`、`Volume Variation`
  - 指标层输出必须为确定性结果（同输入+同参数 -> 同输出）。
- 非目标 / 范围外
  - 不在指标层实现事件特征：金叉/死叉、突破、背离、共振等。
  - 不在本任务实现五维评分、信号决策、交易动作。
  - 不在本任务处理多周期融合决策逻辑。
  - warmup/`NaN` 在评分系统中的处理策略，延后到评分系统任务中讨论与确定。
  - 单 `symbol` 到多 `symbol` 的迁移设计，当前版本不考虑。
- 输入输出或接口
  - 输入：标准 OHLCV DataFrame，至少包含 `timestamp/symbol/open/high/low/close/volume`。
  - 输出：按时间对齐的特征 DataFrame（核心连续值），允许 warmup 区间出现 `NaN`。
  - 指标层实现不允许原地修改输入 DataFrame（immutable input）；`compute(df, params)` 返回新结果。
  - `timestamp` 透传规则：
    - 指标层不修改输入 `timestamp` 列值与顺序，不新增/删除时间行。
    - 输出 `feature_df` 的时间轴与输入 `bars_df.timestamp` 一一对应，再由 `feature_pipeline` 按同一 `timestamp` 合并。
    - 时间语义错误（缺失、重复、不可解析、非 UTC 口径）归类为输入错误并 fail-fast。
  - 多输出指标命名规则：先生成参数前缀 `{indicator}_{参数1_参数2}`，再拼接语义后缀。
    - 例如 `MACD(12,26,9)` 输出：
      - `macd_12_26_9_line`
      - `macd_12_26_9_signal`
      - `macd_12_26_9_hist`
    - 其它多输出指标后缀字典（v1）：
      - `bollinger`：`mid/up/low`
      - `kd`：`k/d/j`
      - `dmi`：`plus_di/minus_di/adx`
  - 统一接口约束（指标族实现必须满足）：
    - `validate_params(params)`：校验参数合法性。
    - `compute(df, params)`：执行计算并返回 `pd.DataFrame`（单列指标也必须返回 DataFrame）。
    - 元信息字段：`name/category/required_columns/output_columns/warmup_bars`。
  - 参数传递采用“同族多实例”机制：通过 `indicator_plan` 声明实例配置，运行时由 `registry` 分发到对应指标族。
    - v1 中 `indicator_plan` 单条配置包含 `instance_id + family + params` 三个字段。
    - `instance_id` 在同一 `indicator_plan` 内必须唯一，用于运行追踪与结果定位。
    - 同一 `indicator_plan` 内禁止重复 `family + params` 组合，避免重复计算。
  - 列命名规范强制为 `{indicator}_{参数1_参数2}`，必须包含参数，不允许别名或自定义缩写。
  - 参数顺序规范：
    - 列名参数顺序由各指标族定义的 `param_order` 决定，不依赖 `params` 字典顺序或字母排序。
    - 列名生成前先完成参数解析与默认值填充，确保命名稳定可复现（例如 `macd_12_26_9`）。
    - 参数字符串化需统一口径：
      - 若参数为浮点且小数部分为 `0`，按整数输出（例如 `2.0 -> 2`）。
      - 若参数为浮点且有小数，保留两位小数（例如 `2.345 -> 2.35`，`2.3 -> 2.30`）。
      - 浮点参数在列名中保留小数点（例如 `bollinger_20_2.30_mid`）。
      - 禁止不同字符串化方式导致列名漂移。
  - 单 `symbol` 计算口径：
    - `model_df` 主键为 `timestamp`（UTC、bar open time），且必须 `非空 + 唯一 + 升序`。
    - `symbol` 保留为普通列（全列同值），不进入主键。
    - `interval` 作为普通列或元信息（全列同值），不进入主键。
    - 若输入 `bars_df` 出现重复 `timestamp`，必须立即报错并中止（fail-fast）。
    - `feature_pipeline` 合并指标结果时必须按 `timestamp` 执行 `left join`，且不得改变 `bars_df` 行数。
  - 报错规范（v1）：
    - 错误消息格式：`XTR018::<ERROR_CODE>::<detail>`
    - 计划配置错误：
      - `PLAN_MISSING_FIELD`：`indicator_plan` 缺少 `instance_id/family/params`
      - `PLAN_DUPLICATE_INSTANCE_ID`：`instance_id` 重复
      - `PLAN_DUPLICATE_FAMILY_PARAMS`：`family + params` 组合重复
      - `PLAN_UNKNOWN_FAMILY`：未注册的指标族
    - 参数错误：
      - `PARAM_MISSING_REQUIRED`：缺少必需参数
      - `PARAM_INVALID_TYPE`：参数类型错误
      - `PARAM_OUT_OF_RANGE`：参数越界
    - 输入数据错误：
      - `INPUT_MISSING_COLUMN`：缺少 OHLCV 必需列
      - `INPUT_TIMESTAMP_DUPLICATE`：`timestamp` 重复
      - `INPUT_TIMESTAMP_INVALID`：`timestamp` 无法解析或非 UTC 口径
    - 输出冲突错误：
      - `OUTPUT_COLUMN_CONFLICT`：同次运行生成重复列名

## Design
- 核心思路与架构
  - 采用“指标计算层（Feature）与事件判定层（Signal）解耦”设计。
  - `feature_engine` 只产出指标值；`signal_engine` 再消费这些值生成派生事件与信号。
- 数据/接口/模型
  - 目录结构（本任务相关）：
    - `src/xtrader/strategies/feature_engine/indicators/trend/`
    - `src/xtrader/strategies/feature_engine/indicators/oscillator/`
    - `src/xtrader/strategies/feature_engine/indicators/volatility/`
    - `src/xtrader/strategies/feature_engine/indicators/volume/`
  - 指标实例化规则：
    - `EMA` 为指标族，`ema_12`/`ema_26` 为参数实例产物，不拆分为多个指标族。
    - 列命名严格遵循 `{indicator}_{参数1_参数2}`（例如 `ema_12`、`macd_12_26_9`），不允许别名。
    - 参数顺序由指标族固定声明：
      - `ema`: `("period",)`
      - `macd`: `("fast", "slow", "signal")`
      - `bollinger`: `("period", "std")`
- 运行组织方式：
    - 增加指标注册中心（`registry`），按 `family` 查找指标实现。
    - 增加实例计划（`indicator_plan`），按顺序执行并合并特征列。
    - `feature_pipeline` 负责统一拼接：`model_df = concat([bars_df, feature_df], axis=1)`。
    - 评分系统后续从 `model_df` 读取约定特征列（如 `ema_12/rsi_14/macd_*`）；本任务只保证数据契约，不实现评分逻辑。
    - 时间列处理为“只读透传”：指标计算只追加特征列，不改动时间轴。
  - `indicator_plan` 示例（v1）：
    - `{"instance_id":"ema_12","family":"ema","params":{"period":12}}`
    - `{"instance_id":"ema_26","family":"ema","params":{"period":26}}`
    - `{"instance_id":"macd_12_26_9","family":"macd","params":{"fast":12,"slow":26,"signal":9}}`
- 风险与权衡
  - 若在指标层混入事件特征，短期看实现快，但长期会导致职责耦合、重复逻辑和维护成本上升。
  - 将事件逻辑放入 `signal_engine`，短期需要多一层处理，但边界清晰、回测解释更一致。

## Acceptance
- 代码结构验收
  - 指标分类目录存在，且后续指标文件按“一个指标族一个文件”落位。
- 功能验收
  - v1 指标族可对标准 OHLCV 输入计算并输出连续值。
  - 参数化实例生效（如 `EMA(12)` 与 `EMA(26)` 输出不同列值）。
  - 多输出指标列名遵循“参数前缀 + 语义后缀”规则；`MACD(12,26,9)` 必须输出 `macd_12_26_9_line/signal/hist`。
  - 非 MACD 多输出后缀遵循 v1 字典：
    - `bollinger_*_*_mid/up/low`
    - `kd_*_*_*_k/d/j`
    - `dmi_*_*_plus_di/minus_di/adx`
  - 不同指标族均可通过统一接口被调度（`validate_params/compute` 调用路径一致）。
  - `compute` 返回类型统一为 `DataFrame`（包含单列指标）。
  - `indicator_plan` 可同时声明同族多实例，并在一次执行中产出多组结果列。
  - `indicator_plan` v1 要求 `instance_id + family + params`，且 `instance_id` 在计划内唯一。
  - 同一计划中重复 `family + params` 必须报错，不允许重复计算。
  - 所有输出列名必须包含参数且符合 `{indicator}_{参数1_参数2}`；不得出现别名列。
  - 列名参数顺序必须遵循各指标族 `param_order`，且默认参数需参与命名。
  - 浮点参数字符串化满足：整数化优先；非整数统一保留两位小数。
  - 浮点参数列名保留小数点（例如 `bollinger_20_2.30_mid`）。
  - 输入 DataFrame 不被原地修改；指标计算结果以新 DataFrame 返回。
  - 输入与输出 `timestamp` 保持严格一致（值、顺序、行数）。
  - `feature_pipeline` 能产出供评分系统消费的 `model_df`，并保持时间索引对齐。
  - 单 `symbol` 模式下，`model_df` 的 `timestamp` 满足非空、唯一、升序；合并后行数与原始 `bars_df` 一致。
  - 若输入存在重复 `timestamp`，流程必须报错而不是自动去重或覆盖。
  - 所有失败路径必须返回规范化错误码（`XTR018::<ERROR_CODE>::...`）。
- 质量验收
  - 指标层不包含金叉/死叉等事件逻辑。
  - 核心指标具备单元测试，覆盖参数合法性、warmup 行为、输出列完整性、关键样例值。
