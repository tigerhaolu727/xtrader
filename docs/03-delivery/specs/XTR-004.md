# 多交易所可扩展K线下载与本地存储框架（Bitget优先） (XTR-004)

## Intent
当前项目已具备 Bitget K 线拉取能力，但缺少“可长期运行的数据下载程序”。本任务目标是定义一套可扩展、可增量、可重跑、可校验的数据下载方案：
- 先支持 Bitget（现货/合约）。
- 支持 3-5 年长区间下载与断点续跑。
- 支持向前/向后增量补数与重置重下。
- 统一时间规范（UTC、毫秒）并校验 K 线时间对齐，避免数据混淆。

## Requirement
- 功能目标
  - 提供统一下载入口：按交易所、市场类型、交易对、K 线周期、起止时间下载 K 线。
  - 支持下载模式：
    - `full`：按给定区间全量下载（允许覆盖已有分区）。
    - `append`：只补本地最大时间之后的数据。
    - `prepend`：只补本地最小时间之前的数据。
    - `repair`：按目标区间扫描缺口并补洞。
    - `reset`：删除目标数据后重下。
  - 产出本地标准化数据文件与元数据索引，避免不同交易所/市场/周期数据混放。
  - 支持失败重试、限速控制和分页下载，能覆盖 3-5 年历史数据。
  - 同一任务可重复执行，结果应幂等（重复运行不产生重复 K 线）。
- 非目标 / 范围外
  - 本任务不实现 WebSocket 实时推送落盘。
  - 本任务不覆盖多交易所完整适配，仅落地 Bitget Provider 与扩展接口。
  - 本任务不引入数据库（先用本地文件系统）。
- 输入输出或接口
  - 输入参数（CLI/API）至少包含：
    - `exchange`（首期仅 `bitget`）
    - `market_type`（`spot`/`linear_swap`/`inverse_swap`）
    - `symbol`（如 `BTCUSDT`）
    - `interval`（与内部 `CandleInterval` 对应）
    - `start_time`、`end_time`
    - `mode`（`full|append|prepend|repair|reset`）
    - `--force-reset`（危险操作显式确认）
  - 输出：
    - 分区 K 线文件（建议 Parquet）
    - 覆盖范围索引（最小/最大时间、记录数、更新时间）
    - 运行日志与验证报告（缺口、重复、异常行）

## Design
- 核心思路与架构
  - `KlineProvider` 抽象层：定义统一拉取接口，屏蔽交易所差异。
  - `BitgetKlineProvider`：封装 Bitget 的 recent/history 接口与分页策略。
  - `SyncPlanner`：根据本地覆盖范围与目标区间生成下载任务片段（可断点续跑）。
  - `KlineStorage`：负责分区写入、去重合并、原子替换。
  - `IntegrityChecker`：做时间对齐、重复检测、缺口检测。
- 数据/接口/模型
  - 统一 K 线记录字段：
    - `exchange`, `market_type`, `symbol`, `interval`
    - `open_time_ms`（UTC 毫秒）
    - `close_time_ms`（UTC 毫秒）
    - `open`, `high`, `low`, `close`, `volume`, `turnover`
    - `source`（history/recent）
  - 本地目录建议：
    - `data/klines/exchange={exchange}/market={market_type}/symbol={symbol}/interval={interval}/year={YYYY}/month={MM}/part-*.parquet`
    - `data/klines_index/{exchange}.{market}.{symbol}.{interval}.json`
  - 去重主键：`(exchange, market_type, symbol, interval, open_time_ms)`。
  - 时间边界统一采用 `[start, end)`（左闭右开），避免重复边界 K 线。
- 风险与权衡
  - 风险：长区间下载受 API 限频影响，任务耗时高。
    - 对策：窗口分页 + 退避重试 + checkpoint。
  - 风险：不同接口返回“最新未完结K线”可能污染历史。
    - 对策：`history` 优先用于历史回填；`recent` 数据可配置过滤未闭合 K 线。
  - 风险：本地已有数据可能跨版本 schema 不一致。
    - 对策：索引写入 `schema_version` 并在运行前检查迁移策略。
  - 权衡：首期不用 DB，降低复杂度；代价是大规模查询能力一般，但符合当前下载归档目标。

## Acceptance
- 在 `bitget + spot + BTCUSDT + 1m` 场景下，可完成 >= 1 年区间下载，数据连续无重复。
- 支持 `append/prepend` 双向增量，重复运行不产生重复记录。
- `reset` 模式可安全删除目标分区并完整重建。
- 输出目录结构能明确区分交易所、市场、交易对、周期、年月。
- 所有写入记录满足：
  - `open_time_ms` 为 UTC 毫秒整数
  - `open_time_ms % interval_ms == 0`
  - 区间边界遵循 `[start, end)`
- 校验报告可输出：新增条数、重复条数、缺口数量、起止覆盖范围。

## Notes
- 首期仅实现 Bitget Provider，但抽象层不得与 Bitget 强耦合。
- 下载程序建议提供 CLI（例如 `python -m xtrader.data.download_klines ...`）。
- 未来扩展方向：
  - 多交易所 Provider（Binance/OKX 等）
  - 并发下载调度
  - 增量调度任务（cron/airflow）
