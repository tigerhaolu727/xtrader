# Network connectivity verification (XTR-002)

## Intent
最近在运行 `tests/integration/test_bitget_client_live.py` 时，由于沙箱网络受限或代理配置问题，导致无法确认 Bitget API 是否可访问。用户希望了解当前会话是否具备直连外网能力，好判断后续集成测试与调试步骤。该任务旨在以最小步骤验证当前开发环境对公开 HTTP 端点（尤其是 Bitget REST API）的网络连通性，并将结果形成记录。

## Requirement
- 功能目标
  - 通过脚本或命令行向 Bitget 公共 REST 端点（如 `/api/spot/v1/public/time`）发起 HTTP 请求，观察是否返回 2xx 响应。
  - 若 Bitget 端点不可达，再使用通用公网服务（如 `https://ifconfig.me`）进行补充验证，明确是全局封锁还是特定域名受限。
  - 记录执行命令、时间戳及结果，方便后续排障或提交给验证守门人。
- 非目标 / 范围外
  - 不在本任务中实现新的客户端功能或修改业务逻辑；仅做网络探测。
  - 不尝试绕过组织策略（如必须走代理才能联网）。若受限，记录事实即可。
- 输入输出或接口
  - 输入：现有环境变量（`BITGET_HTTP_PROXY/BITGET_HTTPS_PROXY` 等）、命令行脚本。
  - 输出：HTTP 响应状态、错误栈或说明，将摘要写入 Validation 文档。

## Design
- 核心思路与架构
  - 使用 `python -c`（内置 `httpx`）或 `curl` 直接发起 GET 请求，避免依赖项目业务代码，降低干扰。
  - 先尝试 Bitget 公共 API；若失败，再请求一个通用公共 API 比较网络行为。
- 数据/接口/模型
  - 仅解析 JSON 响应中关键字段（如服务器时间），或在失败时记录异常类型。
  - 将命令与输出粘贴进验证记录，作为 Evidence。
- 风险与权衡
  - 沙箱可能继续禁止外出连接，测试只能得出“仍被阻断”的结论。
  - 若代理变量被设置但不可用，命令需显式读取/打印当前代理值以协助排查。

## Acceptance
- 成功条件：
  - 至少一个对 Bitget 公共端点的 HTTP 请求获得 2xx 响应，并展示响应片段；若 Bitget 不可达，但其他公网 API 可访问，也应给出对比说明。
  - 如全部请求失败，需在 Validation Execution Log 中记录命令、失败栈及推测原因（如“端口禁止外连”）。
- 产出：
  - 更新 `docs/03-delivery/validation/XTR-002.md` 的 Planned Validation/Execution Log，填写所执行命令和结果。
  - 在 Session Notes 中添加本任务链接（待会话结束前完成）。

## Notes
- 依赖：系统 `python`、`httpx` 已在 `pyproject` 中，命令可复用仓库虚拟环境。
- 如需代理，可读取 `.env` 或环境变量，必要时打印但勿泄漏敏感凭证。
- 若后续用户要求更深入验证（如运行 `pytest` 集成用例），需在此 Spec 基础上扩展示例。
