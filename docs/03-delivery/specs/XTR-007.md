# 统一因子抽象与可扩展输入框架 (XTR-007)

## Intent
当前项目已完成数据获取、重采样底座、防泄漏守门与对账抽检，下一阶段进入因子研发与策略研发。为避免“因子各写各的、输入口径不统一、后续难以回测与复用”的问题，本任务目标是建立统一因子抽象与可扩展输入框架，使因子研发、评估、策略回测和后续应用共享同一接口协议。

## Requirement
- 功能目标
  - 定义统一因子接口，所有因子必须遵循同一输入输出规范。
  - 定义标准运行上下文 `FactorContext`，至少包含：`as_of_time`、`universe`、`inputs`、`params`、`meta`。
  - 定义标准输出对象 `FactorResult`，至少包含：`timestamp`、`symbol`、`factor_id`、`factor_version`、`value`。
  - 定义因子元信息 `FactorSpec`，至少包含：`factor_id`、`version`、`required_inputs`、`warmup_bars`、`params_schema`、`output_schema`。
  - 因子必须显式声明依赖输入（如 `kline.5m`、`kline.1h`、`funding.8h`），框架按声明加载与校验数据。
  - 因子运行必须只通过 `FactorContext.inputs` 读取数据，禁止在因子内部直接拉取外部 API 或读取任意文件。
  - 支持输入源扩展（Funding、OI、Orderbook、Onchain、News），并保证旧 Kline 因子在新增输入源后无需修改即可运行。
  - 提供因子注册与调度能力，支持批量执行、版本化产物落盘与可复现回放。
  - 提供统一评估对接接口，支持 IC/分层收益/换手/成本后收益/稳定性评估模块复用。
- 非目标 / 范围外
  - 本任务不实现完整策略执行引擎（下单、风控、交易路由）。
  - 本任务不实现全部输入源采集器，仅定义扩展接口与接入协议。
  - 本任务不引入分布式调度系统，先满足单机研究流程。
- 输入输出或接口
  - 输入：
    - 标准数据视图 `DataView`（含 `name`、`schema_version`、`df`）
    - 因子运行上下文 `FactorContext`
    - 因子参数 `params`
  - 输出：
    - 标准因子结果 `FactorResult`
    - 因子运行元数据与评估对接所需索引信息

## Design
- 核心思路与架构
  - 分层架构：`InputAdapter` -> `FactorContextBuilder` -> `FactorEngine` -> `FeatureStore` -> `Evaluator/Backtest`。
  - 统一协议对象：
    - `FactorSpec`：描述因子契约与依赖。
    - `BaseFactor`：定义 `spec()` 与 `compute(context)` 抽象方法。
    - `FactorResult`：定义统一输出 schema 与质量标记。
  - 注册调度机制：
    - `FactorRegistry` 负责注册、发现、版本路由。
    - `FactorEngine` 负责依赖输入校验、warmup 处理、批量执行、异常收敛。
  - 防泄漏约束：
    - 因子只能访问 `as_of_time` 之前可见数据。
    - 信号执行延迟与未来函数检查由框架守门，不依赖单因子自觉实现。
- 数据/接口/模型
  - `FactorContext.inputs` 使用键空间规范：
    - `kline.5m`、`kline.15m`、`funding.8h`、`oi.5m` 等。
  - `DataView` 统一包含 `schema_version`，支持输入结构演进与兼容检查。
  - `FactorResult.values` 最小字段：
    - `timestamp`、`symbol`、`factor_id`、`factor_version`、`value`
  - 产物标识规范：
    - `factor_id + factor_version + params_hash` 作为唯一计算身份，支持复现与缓存。
- 风险与权衡
  - 风险：输入源扩展后时间对齐与缺失处理复杂度上升。
    - 对策：统一在 `InputAdapter/ContextBuilder` 层处理，不下放到单因子。
  - 风险：因子接口若约束不足，后续策略回测难复用。
    - 对策：严格执行统一 schema 与注册校验。
  - 权衡：先实现最小可用抽象（Kline 优先），再渐进接入 Funding/OI 等输入源。

## Acceptance
- 至少 2 个 Kline 因子（例如 `mom_20`、`volatility_20`）按统一接口接入并运行成功。
- 因子输出必须满足标准 schema，且可直接被统一评估模块消费。
- 在仅 Kline 输入场景可运行；新增一种非 Kline 输入定义后，旧因子无需改动仍可运行。
- 因子运行结果满足可复现要求（同输入、同参数、同版本输出一致）。
- 防泄漏守门通过（无未来数据访问，执行延迟规则可校验）。
- 因子结果可被策略回测模块直接读取并完成一次端到端样例回测。
