# Strategy Runtime Core v1（Backtest-first） (XTR-019)

## Intent
- 当前策略研发已进入“声明式配置 + 多周期 + 评分规则 + 动作输出”的阶段，但运行编排仍以脚本拼装为主，存在以下问题：
  - 配置、评分规则、策略规则与物理特征列名耦合，参数变动容易不同步；
  - 回测链路可运行，但缺少统一的 precompile（预编译）与 fail-fast 校验机制；
  - 后续实盘链路需要复用同一套策略运行管理逻辑，避免回测/实盘双轨分叉。
- 本任务目标是先在回测场景落地一套统一 Strategy Runtime Core（Backtest-first），并确保后续可平滑迁移到实盘执行适配器。

## Requirement
### 功能目标
- 引入统一运行编排入口（Runtime Core），以声明式配置驱动策略闭环执行。
- 支持 `FeatureRef` 与 `feature_catalog` 自动生成，并在运行前执行 precompile 校验。
- 支持 trial 级运行（单配置或参数寻优候选配置），确保每个 trial 的配置与映射隔离。
- 在回测链路中接入 Runtime Core，输出标准 run 产物，含可追溯配置与编译证据。
- 保持策略接口不变（`BaseActionStrategy.generate_actions`），兼容评分型策略与直判型策略。

### 非目标 / 范围外
- 不在本任务实现 OMS/EMS 实盘执行细节。
- 不在本任务实现组合层资金分配与加减仓管理。
- 不在本任务实现完整参数优化算法（遗传/贝叶斯）；仅定义并支持 trial 输入输出契约。

### 输入输出 / 接口
- 输入：
  - 声明式策略配置（JSON，含 `indicator_plan_by_tf / scoring_rules / fusion_rules / risk_rules`）。
  - 历史 K 线数据源（回测模式）。
  - 可选参数模板与候选 trial 列表。
- 输出：
  - 标准动作表（`timestamp/symbol/action/size/stop_loss/take_profit/reason`），其中 `stop_loss/take_profit` 统一为绝对价格位（float64）。
  - `reason` 字段在 v1 统一承载 `reason_code`（枚举值）；可选 `reason_detail` 仅用于补充解释，不参与主逻辑判断。
  - `action` 枚举固定为 `ENTER_LONG/ENTER_SHORT/EXIT/HOLD`（v1 不启用 `REVERSE`）。
  - `BUY/SELL/HOLD` 到 `action` 映射基于“当前持仓”执行：`BUY+空仓=>ENTER_LONG`、`SELL+空仓=>ENTER_SHORT`、`BUY+空头=>EXIT`、`SELL+多头=>EXIT`、其他=>`HOLD`。
  - 同一 bar 仅允许一个动作，不允许同 bar 同时执行“平仓+开仓”反手。
  - `size` 统一语义为账户权益占比（`0~1`）的仓位指令；方向由 `action` 决定（`size` 不承载多空方向）。
  - 约束：`ENTER_LONG/ENTER_SHORT` 必须 `size>0`；`EXIT/HOLD` 必须 `size=0`；非法值（负数、`>1`、动作与 size 冲突）应 fail-fast。
  - 策略内部可基于 `%` 或 `ATR` 计算止损止盈，但必须在动作输出前换算为绝对价格；执行层不做二次解释。
  - 回测结果与 run 产物（manifest + parquet/chunks）。
  - trial 级 `resolved_config`、`feature_catalog`、`precompile_report`、`config_hash/catalog_hash`。
  - 面向 Report Viewer 的结构化数据文件（不产出独立 report 文档）。

## Design
### 核心思路与架构
- 采用“统一 Runtime Core + 执行适配器”设计：
  - Runtime Core：读取配置、预编译、特征构建、策略调用、产物落盘。
  - Backtest Executor Adapter：调用现有 `run_event_driven_backtest`。
  - Live Executor Adapter：后续扩展（本任务只预留接口）。

### 模块拆分（v1）
1. `ConfigLoader`
   - 读取并校验声明式配置。
2. `TrialResolver`
   - 将基础配置解析为一个或多个 trial 配置（默认单 trial）。
3. `PrecompileEngine`
   - 解析 `indicator_plan_by_tf`；
   - 生成 `feature_catalog`（`FeatureRef -> physical_col`）；
   - 校验评分/策略规则引用；
   - 输出结构化编译报告（成功/失败 + 错误码）。
4. `FeatureBuilder`
   - 按配置计算多周期特征并对齐执行周期。
5. `StrategyInvoker`
   - 调用策略 `generate_actions` 获取动作表。
6. `ExecutorAdapter(Backtest)`
   - 执行动作撮合与风险闭环，产出 trades/equity/summary/diagnostics。
7. `ArtifactWriter`
   - 在现有 run 产物基础上补充 trial 编译证据文件；
   - 仅输出结构化数据文件供 Report Viewer 消费，不输出独立 report 文件。

### 数据/接口/模型
- 配置格式（v1）：
  - 主格式固定为 `JSON`；
  - 建议文件命名 `*.strategy.json`；
  - 顶层必须包含 `schema_version`，且当前仅接受 `xtr_runtime_v1`；
  - 历史 `path/search_space` 风格不再作为 v1 主口径；
  - 顶层必填字段：
    - `schema_version`
    - `strategy_id`
    - `execution_timeframe`
    - `timeframes`
    - `indicator_plan_by_tf`
    - `signal_rules`
    - `risk_rules`
  - 顶层可选字段：
    - `scoring_rules`
    - `fusion_rules`
    - `trial_config`
    - `warn_policy`
    - `metadata`
  - 默认值策略（v1）：
    - 未提供 `trial_config` 时按单 trial（baseline）执行；
    - 未提供 `scoring_rules` 时按“非评分型策略”路径执行。
    - 未提供 `warn_policy` 时默认 `record_only`。
  - 未知字段策略（v1）：
    - 顶层与关键子结构默认 `additionalProperties=false`；
    - 出现未知字段应 fail-fast 报错；
    - 需要扩展字段时统一放入 `metadata.extensions`。
- `FeatureRef` 规范：`{timeframe}.{instance_id}.{output_key}`。
- `feature_catalog` 字段最小集：
  - `feature_ref`, `physical_col`, `timeframe`, `family`, `instance_id`, `resolved_params`, `params_hash`。
- 关键接口（v1 冻结）：
  - `RuntimeCore.run(config, data_source, mode=\"backtest\") -> RuntimeRunResult`
  - `PrecompileEngine.compile(config, trial_selector=None) -> PrecompileResult`
  - `RuntimeRunResult` 最小必含字段：
    - `run_id`, `status`, `artifacts_root`, `manifest_path`, `summary_path`, `error_code?`, `error_message?`。
  - `PrecompileResult` 最小必含字段：
    - `status`, `catalog_path`, `report_path?`, `error_code?`, `error_message?`。
  - 兼容性规则（v1）：
    - 上述接口签名与最小字段在 v1 期间不得删除/改名/变更语义；
    - 仅允许向后兼容新增字段；
    - 任何破坏性变更必须升级 `schema_version`（例如 `xtr_runtime_v2`）。

### 规则细化（与运行管理文档对齐）
1. `output_key` 约束：
   - 单输出指标固定为 `value`。
   - 多输出指标采用族内后缀字典（v1）：
     - `macd`: `line/signal/hist`
     - `bollinger`: `mid/up/low`
     - `kd`: `k/d/j`
     - `dmi`: `plus_di/minus_di/adx`
2. `FeatureRef` 稳定性约束：
   - 参数寻优与调参场景优先保持 `instance_id` 稳定；
   - params 改变允许 `physical_col` 改变，但 `FeatureRef` 不应变化。
3. 运行期访问约束：
   - 评分与策略规则统一通过 `FeatureAccessor.get(feature_ref)` 访问特征；
   - 禁止在规则表达中直接索引 `physical_col`。
4. trial 产物最小集：
   - `resolved_config.json`
   - `feature_catalog.json`
   - `precompile_report.json`
   - `config_hash`, `catalog_hash`, `trial_id`
5. 缓存一致性约束：
   - 特征缓存 key 必须包含 `symbol + timeframe + data_range + family + params_hash + code_version + data_version`。
   - `params_hash` 计算规范（v1）：
     - 算法：`SHA-256`；
     - 输出格式：`sha256:<64hex>`；
     - 输入：单指标 `params` 的规范化 JSON（Canonical JSON）；
     - 浮点先做规范化：整数值写作整数（如 `2`），非整数保留两位（如 `2.30`）。
6. `trial_config` 口径（v1）：
   - 采用 `mode=scenarios`；
   - 每个 scenario 使用 `changes` 对基线配置做局部覆盖；
   - 覆盖定位采用 `timeframe + instance_id`（不使用 path/search_space 字符串路径）。
   - `trial_id` 在同一配置内必须唯一；
   - 单个 scenario 内 `changes` 的覆盖目标（`timeframe + instance_id`）不得重复；
   - `changes` 采用原子生效：任一 change 非法则整个 scenario 失败，不允许部分生效。
7. 配置一致性约束（v1）：
   - `execution_timeframe` 必须属于 `timeframes`；
   - `indicator_plan_by_tf` 的所有 key 必须属于 `timeframes`；
   - 同一 timeframe 内 `instance_id` 必须唯一；
   - 同一 timeframe 内禁止重复 `family + params` 组合。
8. `risk_rules` schema 约束（v1）：
   - `position_size`：
     - `mode` 仅允许 `fixed_fraction`；
     - 必填 `value`，取值范围 `(0,1]`。
   - `stop_loss`：
     - `mode` 仅允许 `atr_multiple`；
     - 必填 `n`（`int >= 1`）与 `k`（`float > 0`）。
   - `take_profit`：
     - `mode` 仅允许 `rr_multiple`；
     - 必填 `rr`（`float > 0`）。
   - 出现未知 `mode`、字段缺失、类型不匹配或越界值时必须 fail-fast；
   - 上述配置错误统一命中 `PC-CFG-003`。
9. `trial_config` 对 `risk_rules` 的覆盖约束（v1）：
   - v1 不支持通过 scenario `changes` 覆盖 `risk_rules`；
   - scenario 仅允许覆盖指标参数（`timeframe + instance_id + params`）；
   - 尝试覆盖 `risk_rules` 必须 fail-fast（`PC-TRI-001`）。
10. precompile 错误码规范（v1）：
   - 编码格式：`PC-{DOMAIN}-{NNN}`（例如 `PC-CFG-001`）；
   - `DOMAIN` 取值：`CFG/TFM/DUP/OUT/REF/TRI/SYS`；
   - 严重级别：`ERROR/WARN`；
   - `ERROR` 永远阻断运行；
   - `WARN` 由 `warn_policy` 决定：
     - 默认：`warn_policy=record_only`（仅记录，不阻断）；
     - 可选：`warn_policy=error`（将 `WARN` 升级为失败，阻断运行，适用于 CI/批量试验）；
   - `warn_policy` 配置入口与优先级（v1）：
     - 配置文件顶层字段：`warn_policy`；
     - CLI 覆盖参数：`--warn-policy`；
     - 生效优先级：`CLI > config > default(record_only)`；
     - 同一 run 仅允许一个生效值，不支持 per-trial 单独设置；
     - 生效值必须同时写入 `precompile_report` 与 `run_manifest`；
     - 非法取值或类型错误必须 fail-fast，并命中 `PC-CFG-003`。
   - `precompile_report` 必须输出：`warn_policy`、`warn_count`、`warn_codes`；
   - `run_manifest` 必须输出：`warn_policy`、`warn_count`、`warn_codes`（用于运行审计）；
   - 错误载荷最小字段：
     - `code`
     - `message`
     - `stage`（`config_load/schema/precompile`）
     - `severity`
     - `path`（JSON 路径）
     - `timeframe`（可空）
     - `instance_id`（可空）
     - `feature_ref`（可空）
     - `suggestion`（可空）
   - v1 首批错误码字典：

| code | domain | trigger |
| --- | --- | --- |
| `PC-CFG-001` | `CFG` | 必填字段缺失 |
| `PC-CFG-002` | `CFG` | 未知字段（不在 `metadata.extensions`） |
| `PC-CFG-003` | `CFG` | 字段类型不匹配 |
| `PC-TFM-001` | `TFM` | `execution_timeframe` 不在 `timeframes` |
| `PC-TFM-002` | `TFM` | `indicator_plan_by_tf` key 不在 `timeframes` |
| `PC-DUP-001` | `DUP` | 同一 timeframe 内 `instance_id` 重复 |
| `PC-DUP-002` | `DUP` | 同一 timeframe 内 `family+params` 重复 |
| `PC-OUT-001` | `OUT` | `output_key` 非法（未命中单/多输出规范） |
| `PC-REF-001` | `REF` | 规则引用的 `FeatureRef` 无法解析 |
| `PC-TRI-001` | `TRI` | scenario `changes` 目标 `timeframe+instance_id` 不存在 |
| `PC-TRI-002` | `TRI` | scenario 冲突（重复 `trial_id` 或重复 `timeframe+instance_id`） |
| `PC-SYS-001` | `SYS` | precompile 内部异常（兜底） |
11. 多时间周期对齐语义（v1）：
   - `timestamp` 统一语义为 `bar open time`；
   - 可用性规则：源周期 bar 仅在 `timestamp + source_timeframe` 之后可被使用；
   - 对齐规则：对执行时刻 `t_exec` 做 as-of 对齐时，仅可选取满足 `source_timestamp + source_timeframe <= t_exec` 的最近一条；
   - 同周期规则：当 `source_timeframe == execution_timeframe` 时，仍遵守上述可用性规则，等价为 1-bar lag；
   - v1 约束：仅支持 `source_timeframe >= execution_timeframe`；
   - 缺失值策略：对齐缺失统一保留 `NaN`，不做前向填充；
   - 右标签兼容：若输入为 right-label 口径，入库/预处理阶段必须先转换为 open-time 语义再进入特征链路。
12. `FeatureAccessor` 行为语义（v1）：
   - `FeatureRef` 不存在（未注册/未编译通过）视为配置错误，必须抛出运行时错误并中止；
   - `FeatureRef` 存在但当前值为 `NaN`，`get(feature_ref)` 返回 `NaN`，不主动抛错；
   - 评分/信号层收到 `NaN` 时按规则“未满足/0 分”处理，不得将 `NaN` 隐式替换为数值；
   - 仅当关键运行字段缺失（例如执行主键、动作必需字段）时，运行层可中止；
   - `FeatureAccessor` 提供可选 `get_or_default(feature_ref, default)` 仅供非核心辅助逻辑使用，评分主链路禁止使用默认值兜底。
13. trial 调度策略（v1）：
   - 执行前先对全部 trial 完成 precompile；
   - precompile 失败 trial 不进入执行阶段；
   - 运行顺序固定为 `trial_config.scenarios` 声明顺序；
   - 默认串行执行（`max_parallel_trials=1`）；
   - 失败策略：
     - 单 trial：`abort`（直接失败退出）；
     - 多 trial：`continue`（记录失败 trial，继续执行其余 trial）；
   - run summary 必须包含 `succeeded_trial_ids`、`failed_trial_ids`、`skipped_trial_ids`。
   - run 终态与退出码（v1）：
     - `run_status=SUCCESS`：所有进入执行的 trial 成功；
     - `run_status=PARTIAL_SUCCESS`：至少一个成功且至少一个失败/跳过；
     - `run_status=FAILED`：无成功 trial（全失败或被阻断）；
     - 单独出现 `signals.status=SKIPPED` 不改变上述 run 终态判定；
     - `exit_code=0` 对应 `SUCCESS`；
     - `exit_code=2` 对应 `PARTIAL_SUCCESS`；
     - `exit_code=1` 对应 `FAILED`。
14. 缓存生命周期策略（v1）：
   - v1 仅支持 run 级共享缓存（同一次 run 内可复用），不支持跨 run 复用；
   - 每个 trial 结束后释放 trial 临时对象；
   - run 结束后清空 run 级缓存，防止跨 run 污染；
   - 优先复用“共享特征集合”，trial 仅增量计算变更部分；
   - 评分/信号/策略判定阶段不引入跨 trial 状态缓存。
15. run 产物目录与 Viewer 对接（v1）：
   - 使用单一 run 根目录：`runs/{run_id}/`；
   - `run_manifest` 保持向后兼容，不删除/重命名已有字段；
   - `manifest_version` 版本门禁：
     - `manifest_version` 必须为整数，且 `>=1`；
     - v1 解析器仅保证对 `manifest_version=1` 的完整兼容；
     - 未来版本必须通过显式版本分支解析，不允许“猜字段”解析。
   - v1 兼容边界：
     - v1 最小必填字段为稳定契约，允许新增字段，但不得删除/重命名/变更语义；
     - 字段弃用采用两阶段：先标记 deprecated 并保留读取，再于下一大版本移除；
     - 对 `manifest_version=1`，若缺失任一 v1 必填字段，Viewer 必须返回 `INVALID_RUN`。
   - `run_manifest` 最小必填字段（v1）：
     - `manifest_version`
     - `run_id`
     - `task_id`
     - `mode`
     - `strategy_id`
     - `schema_version`
     - `execution_timeframe`
     - `timeframes`
     - `symbol`
     - `data_range.start/end`
     - `trial_summary.succeeded_trial_ids/failed_trial_ids/skipped_trial_ids`
     - `trial_summary.total_trials`
     - `run_status`
     - `exit_code`
     - `config_refs.raw_config_path/resolved_config_path/resolved_config_hash`
     - `code_version`
     - `data_version`
     - `artifacts`（关键数据文件路径索引）
   - `run_manifest` 不存配置细节（如指标参数、评分规则、风险规则）；配置唯一事实来源为独立 JSON 文件；
   - 配置文件独立保留：
     - `config/strategy_config.raw.json`（原始输入 JSON，原样保留）
     - `config/strategy_config.resolved.json`（默认值与 scenario 覆盖后的最终配置）
   - trial 证据文件放置于 run 目录下：
     - `precompile/feature_catalog.json`
     - `precompile/precompile_report.json`
  - 回测/策略运行输出结构化数据文件（如 `summary/signals/trades/equity/diagnostics`）供 Report Viewer 打开；
   - 不产出独立的 Markdown/HTML 报告文件。
16. `runs` 单目录输出策略（v1）：
   - 运行产物仅写入 `runs/{run_id}/`；
   - `report` 目录不再作为运行产物输出目标；
   - `runs/{run_id}/` 必须完整落盘，作为长期标准产物目录。
17. Report Viewer 输入契约（v1）：
   - 必需文件（缺一不可）：
     - `run_manifest.json`
     - `artifacts/summary.parquet`
     - `artifacts/signals.parquet`
     - `artifacts/trades.parquet`
     - `artifacts/equity.parquet`
     - `data_snapshot/raw/{timeframe}.parquet`（按周期分文件，至少覆盖 `required_timeframes`）
     - `data_snapshot/resampled/{timeframe}.parquet`（按周期分文件，至少覆盖 `required_timeframes`）
     - `data_snapshot/dataset_index.json`
     - `data_snapshot/snapshot_meta.json`
   - 可选文件（缺失时降级展示）：
     - `artifacts/diagnostics.parquet`
     - `precompile/feature_catalog.json`
     - `precompile/precompile_report.json`
   - 最小 schema（v1）：
     - `artifacts/summary.parquet`（每个 `run_id + trial_id` 一行）：
       - `run_id`(string), `trial_id`(string), `run_status`(string)
       - `total_return`(float64), `max_drawdown`(float64), `sharpe`(float64), `win_rate`(float64)
       - `trade_count`(int64), `skipped_signal_count`(int64), `code_version`(string), `data_version`(string)
     - `artifacts/signals.parquet`（信号/执行结果）：
       - `run_id`(string), `trial_id`(string), `symbol`(string)
       - `signal_time`(datetime64[ns, UTC]), `execution_time`(datetime64[ns, UTC], nullable)
       - `signal`(string), `action`(string), `status`(string:`FILLED|SKIPPED`)
       - `reason_code`(string), `reason_detail`(string, nullable), `skip_reason`(string, nullable)
       - `skip_reason` 枚举：`NO_NEXT_BAR|MARKET_CLOSED|DATA_GAP|RISK_BLOCKED|INVALID_ORDER`
     - `artifacts/trades.parquet`（逐笔成交）：
       - `run_id`(string), `trial_id`(string), `symbol`(string), `side`(string:`LONG|SHORT`)
       - `entry_time`(datetime64[ns, UTC]), `exit_time`(datetime64[ns, UTC])
       - `entry_price`(float64), `exit_price`(float64), `size`(float64)
       - `pnl`(float64), `pnl_pct`(float64), `reason`(string, `reason_code`)
     - `artifacts/equity.parquet`（净值时序）：
       - `run_id`(string), `trial_id`(string), `timestamp`(datetime64[ns, UTC])
       - `equity`(float64), `cash`(float64), `position_value`(float64), `drawdown`(float64)
     - `artifacts/diagnostics.parquet`（评分/信号诊断，可选）：
       - `run_id`(string), `trial_id`(string), `timestamp`(datetime64[ns, UTC]), `symbol`(string)
       - `market_state`(string), `score_total`(float64)
       - `score_breakdown`(string, JSON object string: `{dimension_id: score}`)
       - `weights_applied`(string, JSON object string: `{dimension_id: weight}`)
       - `signal`(string), `action`(string), `reason_code`(string), `reason_detail`(string, nullable)
     - `data_snapshot/raw/{timeframe}.parquet`：
       - `timestamp`, `symbol`, `timeframe`, `open`, `high`, `low`, `close`, `volume`
     - `data_snapshot/resampled/{timeframe}.parquet`：
       - `timestamp`, `symbol`, `timeframe`, `open`, `high`, `low`, `close`, `volume`
     - `data_snapshot/dataset_index.json`：
       - `datasets[]`（每个元素最小字段：`kind`, `timeframe`, `path`, `row_count`, `start`, `end`, `timezone`, `label_semantics`）
     - `data_snapshot/snapshot_meta.json`：
       - `exchange`, `market`, `symbol`, `timeframes`, `required_timeframes`, `source_range`, `resample_rule`, `data_version`
   - Viewer 行为约定：
     - 缺失任一必需文件：标记为 `INVALID_RUN`，不展示主结果面板；
     - 必需文件列缺失或类型不匹配：标记为 `INVALID_RUN`；
     - 时间列必须为 UTC；数值列不得以字符串类型落盘；
     - 缺失可选文件：主结果可展示，对应子面板标记 `NOT_AVAILABLE`；
     - 可选文件 `diagnostics.parquet` 存在但 schema 不匹配：仅 diagnostics 子面板报错，不影响主结果面板。
   - `diagnostics` JSON 字段语义（v1）：
     - `score_breakdown` 与 `weights_applied` 必须为可解析的 JSON 对象字符串；
     - 两个对象的 value 必须为数值类型；
     - `weights_applied` 的权重和应约等于 1（允许浮点误差：`abs(sum-1)<=1e-6`）；
     - JSON 非法或语义不满足时，仅 diagnostics 子面板报错，不影响主结果面板。
   - `signals/trades` 关系约束（v1）：
     - `signals.status=FILLED` 的信号可形成成交；
     - `signals.status=SKIPPED` 不得出现在 `trades.parquet`；
     - `signals.status=SKIPPED` 时 `skip_reason` 必填且必须命中枚举；
     - `signals.status=FILLED` 时 `skip_reason` 必须为空；
     - `summary.skipped_signal_count` 应等于 `signals.status=SKIPPED` 的数量。
   - `dataset_index/snapshot_meta` 枚举与约束（v1）：
     - `dataset_index.datasets[].kind` 仅允许 `raw` 或 `resampled`；
     - `dataset_index.datasets[].label_semantics` 仅允许 `open_time` 或 `right_label`；
     - 若 `label_semantics=right_label`，运行前必须转换为 `open_time` 语义后再进入特征链路；
     - `dataset_index.datasets[].timezone` 固定为 `UTC`；
     - `dataset_index.datasets[].path` 必须为相对 `runs/{run_id}` 的相对路径，禁止绝对路径；
     - `snapshot_meta.required_timeframes` 必须是 `snapshot_meta.timeframes` 的子集，且必须包含 `execution_timeframe`。
18. 哈希与规范化序列化（v1）：
   - 统一算法：`SHA-256`，输出 `sha256:<64hex>`；
   - 规范化序列化（Canonical JSON）：
     - UTF-8 编码；
     - JSON 对象键按字典序排序；
     - 去除非必要空白；
     - 数组保持原顺序（不排序）；
     - `null` 字段保留；
   - 浮点规范化：
     - 整数值写作整数；
     - 非整数保留两位；
   - 哈希定义：
     - `config_hash`：对 `strategy_config.resolved.json` 规范化后计算；
     - `catalog_hash`：对 `feature_catalog.json` 规范化后计算；
     - `params_hash`：对单指标 `params` 规范化后计算；
   - 排除字段：
     - 运行时路径、运行时间、日志与耗时等非语义字段不纳入上述哈希输入；
   - 一致性要求：
     - 同输入跨机器/跨运行哈希必须一致；
     - 参数或配置语义变化必须触发对应哈希变化。
19. `code_version / data_version` 生成口径（v1）：
   - `code_version`：
     - 格式：`git:<full_commit_sha>` 或 `git:<full_commit_sha>+dirty`；
     - 来源：当前仓库 `HEAD` 提交哈希；
     - 当工作区存在未提交改动时追加 `+dirty`。
   - `data_version`：
     - 格式：`data:sha256:<64hex>`；
     - 来源：对 `data_snapshot/dataset_index.json` 与 `data_snapshot/snapshot_meta.json` 的规范化内容计算哈希；
     - 不使用绝对路径作为哈希输入，避免机器差异。
   - 失败策略：
     - 无法生成 `code_version` 或 `data_version` 时，运行必须 fail-fast 并中止。
   - 落盘要求：
     - `run_manifest` 必须包含 `code_version` 与 `data_version`；
     - `artifacts/summary` 中应镜像写入 `code_version` 与 `data_version` 便于下游查询。
20. 非功能验收阈值（v1）：
   - 基准场景 A（冒烟）：
     - 数据规模：`1 symbol + 90天 + 1 trial + 3 timeframes(15m/1h/4h)`；
     - 阈值：总耗时 `<= 60s`，峰值内存 `<= 2GB`。
   - 基准场景 B（标准）：
     - 数据规模：`1 symbol + 1年 + 10 trials + 3 timeframes(15m/1h/4h)`；
     - 阈值：总耗时 `<= 10min`，峰值内存 `<= 6GB`。
   - precompile 阈值：
     - 标准场景下单次 precompile `<= 5s`。
   - 稳定性阈值：
     - 相同输入重复运行 3 次，`run_status` 一致，且 `config_hash/catalog_hash` 一致。
   - 回归阈值：
     - 相对基线版本性能退化超过 `20%` 视为不通过。
   - 验证日志最小字段：
     - `elapsed_ms`, `peak_rss_mb`, `trials_count`, `bars_count`, `status`。
21. 信号到动作映射（v1）：
   - `action` 允许值仅为 `ENTER_LONG/ENTER_SHORT/EXIT/HOLD`；
   - v1 不支持 `REVERSE`；
   - 映射规则（基于当前持仓）：
     - `BUY + 空仓 => ENTER_LONG`
     - `SELL + 空仓 => ENTER_SHORT`
     - `BUY + 空头 => EXIT`
   - `SELL + 多头 => EXIT`
   - 其余 => `HOLD`
   - 同一 bar 同一 symbol 只允许一个动作；禁止“平仓+开仓”在同 bar 合并执行。
22. 执行时序与成交口径（v1）：
   - `signal_time=t`：基于执行周期 bar `t` 收盘后生成信号；
   - `execution_time=t+1`：在下一根执行周期 bar 开盘执行；
   - 成交价口径固定为 `next_bar_open`，v1 不支持同 bar 成交；
   - `stop_loss/take_profit` 从持仓建立后的后续 bar 开始生效，不在入场同一 bar 触发；
   - 若缺失 `t+1` bar（样本尾部），该信号标记为未成交（skipped），并写入原因信息（如 `reason_detail=NO_NEXT_BAR`）。
23. `reason_code` 语义与枚举（v1）：
   - `reason_code` 固定枚举：
     - `ENTRY_SIGNAL_LONG`
     - `ENTRY_SIGNAL_SHORT`
     - `EXIT_SIGNAL`
     - `EXIT_STOP_LOSS`
     - `EXIT_TAKE_PROFIT`
     - `EXIT_RISK_GUARD`
     - `EXIT_TIME_STOP`
     - `HOLD_NO_SIGNAL`
     - `HOLD_WARMUP`
     - `HOLD_DATA_ISSUE`
   - `action` 与 `reason_code` 必须一致：
     - `action=ENTER_LONG/ENTER_SHORT` 仅允许 `ENTRY_*`；
     - `action=EXIT` 仅允许 `EXIT_*`；
     - `action=HOLD` 仅允许 `HOLD_*`。
   - `reason_detail` 为可选自由文本，只做解释，不得用于执行分支判断；
   - 出现未知 `reason_code` 或 `action/reason_code` 不一致时，按配置错误 fail-fast。

### 风险与权衡
- 风险：
  - 规则表达复杂度上升，precompile 错误码设计不清会影响排障效率。
  - 多周期对齐口径若不统一，易引入前视偏差。
- 权衡：
  - v1 先做 Backtest-first，验证契约与编译闭环，再接 live adapter；
  - 优先保证可追溯与 fail-fast，再优化性能。

1. M1：配置与契约底座

   - 产出：配置加载器与契约校验可运行，错误码可稳定命中。

   - 产出：precompile 成功/失败路径稳定，结构化错误载荷完整。
   - 退出门槛：Validation 的 precompile/错误码/trial 编排检查通过（P0 + P1）。
3. M3：执行闭环与交易语义
   - 范围：时序口径（`signal_time=t`/`execution_time=t+1`）、成交价口径、动作/仓位/风控语义、`signals/trades/equity/summary` 闭环。
   - 产出：回测闭环可产出标准结构化文件，跳过信号统计自洽。
   - 退出门槛：Validation 的执行闭环与动作语义检查通过（P0）。
4. M4：运行产物与 Viewer 契约
   - 范围：`runs/{run_id}` 单目录、`run_manifest` 最小字段、`raw/resampled` 快照、`dataset_index/snapshot_meta`、可选文件降级。
   - 产出：Viewer 基于结构化文件可稳定加载并判定 `INVALID_RUN/NOT_AVAILABLE`。
   - 退出门槛：Validation 的产物完整性与 Viewer 契约检查通过（P1）。
5. M5：可追溯与非功能收口
   - 范围：`config_hash/catalog_hash/params_hash`、`code_version/data_version`、性能/稳定性/回归阈值。
   - 产出：可追溯链路闭环，性能指标达到 v1 阈值。
   - 退出门槛：Validation 的可追溯与非功能检查通过（P2）。

### M2 详细实施计划（实施级）
1. M2-A：TrialResolver 从 `single` 扩展到 `scenarios`
   - 目标：
     - 支持 `trial_config.mode=scenarios`；
     - 输出稳定顺序的 trial 列表（按 `scenarios` 声明顺序）；
     - 保持缺省 `trial_config` 仍走 baseline 单 trial。
   - 实施要点：
     - 新增 `ScenarioResolver`（或等价能力）产出 `resolved_config + trial_id`；
     - 在解析阶段统一执行 `trial_id` 唯一性校验；
     - 解析阶段识别 scenario 内重复覆盖目标（`timeframe+instance_id`）并阻断。
   - 错误码：
     - 冲突类统一命中 `PC-TRI-002`；
     - 非法覆盖边界（如覆盖 `risk_rules`）命中 `PC-TRI-001`。
2. M2-B：Scenario `changes` 原子生效与边界门禁
   - 目标：
     - 单 scenario 任一 `change` 非法时，整个 scenario 失败，不允许部分生效。
   - 实施要点：
     - 先在内存副本上完整应用全部 `changes`，全部通过后再提交为该 trial 的 `resolved_config`；
     - 明确 v1 仅允许覆盖指标参数（`timeframe + instance_id + params`）；
     - 覆盖目标不存在、类型不匹配、路径越界均按结构化错误返回。
3. M2-C：Precompile 完整化（成功路径 + 失败路径）
   - 目标：
     - 对每个 trial 稳定产出 `feature_catalog`；
     - 对引用错误、输出键错误、重复冲突等场景稳定命中约定错误码。
   - 实施要点：
     - 补齐 `FeatureRef` 引用校验；
     - 补齐 `output_key` 白名单约束（单输出 `value`，多输出按族内字典）；
     - 补齐错误载荷最小字段（`code/message/stage/severity/path/...`）。
4. M2-D：`warn_policy` 贯穿 precompile 与 run 审计
   - 目标：
     - 实现 `CLI > config > default(record_only)` 优先级；
     - `record_only` 记录不阻断，`error` 升级阻断。
   - 实施要点：
     - 在 run 入口解析并固定单次 run 生效 `warn_policy`；
     - precompile 输出 `warn_policy/warn_count/warn_codes`；
     - run_manifest 同步写入同名审计字段，确保可追溯一致。
5. M2-E：调度门禁与失败策略
   - 目标：
     - 先完成全部 trial precompile，再进入执行调度；
     - 单 trial 失败 `abort`，多 trial 失败 `continue`，并输出失败 trial 清单。
   - 实施要点：
     - 增加 precompile 阶段汇总结果对象（成功/失败 trial 列表）；
     - 调度器仅消费 precompile 成功 trial；
     - run 终态与退出码映射保持 `SUCCESS/PARTIAL_SUCCESS/FAILED` + `0/2/1`。
6. M2-F：测试与证据要求（交付门槛）
   - 自动化测试最小集：
     - 单元测试：scenario 冲突、原子生效、错误码命中、warn_policy 优先级；
     - 集成测试：2~3 个 scenario 的稳定顺序执行与失败策略。
   - 文档证据最小集：
     - Validation Execution Log 必须记录每组命令、结果摘要、证据路径（含 `run_id`）；
     - 至少 1 条正例 + 1 条反例。
7. M2 退出定义（DoD）
   - Validation 中 M2 相关 P0/P1 条目全部有执行证据；
   - `task_guard.py check XTR-019` 通过；
   - 不破坏 M1 已有测试与回测链路。

### M3 详细实施计划（实施级）
1. M3-A：信号时序与执行时序落地
   - 目标：
     - 固化 `signal_time=t`、`execution_time=t+1`；
     - 保持同一 `execution_time` 单动作调度语义。
   - 实施要点：
     - action 调度快照保留 `signal_time/execution_time`；
     - 统一通过 `execution_lag_bars * interval_ms` 计算执行时点；
     - 无效时间戳/缺失关键字段 fail-fast。
2. M3-B：成交价与 SL/TP 生效时序
   - 目标：
     - 成交价固定 `next_bar_open`；
     - `stop_loss/take_profit` 仅在入场后续 bar 生效，不在入场同 bar 触发。
   - 实施要点：
     - 回测执行顺序保持“推进 bar -> 风控评估 -> 执行动作”；
     - 补充边界回归：入场同 bar 不应触发 SL/TP。
3. M3-C：skipped 信号落盘与统计
   - 目标：
     - 未落到可执行 bar 的信号输出 `SKIPPED`；
     - `summary.skipped_signal_count` 与信号明细计数一致。
   - 实施要点：
     - action 快照增加 `status` 与 `skip_reason`；
     - v1 先覆盖 `NO_NEXT_BAR/MARKET_CLOSED/DATA_GAP`；
     - `SKIPPED` 不写入 trades，不单独导致 run 失败。
4. M3-D：signals/trades/equity/summary 一致性约束
   - 目标：
     - 四类产物形成可校验闭环；
     - summary 指标与明细计数（trade/skipped）一致。
   - 实施要点：
     - 写盘前增加最小一致性校验；
     - 不一致时返回结构化错误并阻断交付。
5. M3-E：M3 退出定义（DoD）
   - Validation 的 M3 必跑项（时序/成交价/SLTP/四表一致性）有正反例证据；
   - `task_guard.py check XTR-019` 通过；
   - 不破坏 M2 既有回归结果。

### M4 详细实施计划（实施级）
1. M4-A：统一运行产物目录与最小文件集合
   - 目标：
     - 固化 `runs/{run_id}` 单目录输出；
     - 增补 Viewer 消费的标准化文件路径（`artifacts/*` + `data_snapshot/*`）。
   - 实施要点：
     - 保留现有回测输出（向后兼容），新增标准化镜像产物：
       - `artifacts/summary.parquet`
       - `artifacts/signals.parquet`
       - `artifacts/trades.parquet`
       - `artifacts/equity.parquet`
     - 产物路径统一写入 `run_manifest`，禁止绝对路径。
2. M4-B：快照索引与元信息落盘
   - 目标：
     - 落地 `data_snapshot/raw/{timeframe}.parquet` 与 `data_snapshot/resampled/{timeframe}.parquet`；
     - 输出 `dataset_index.json` 与 `snapshot_meta.json`。
   - 实施要点：
     - `dataset_index` 最小字段覆盖：`kind/timeframe/path/row_count/start/end/timezone/label_semantics`；
     - `snapshot_meta` 最小字段覆盖：`exchange/market/symbol/timeframes/required_timeframes/source_range/resample_rule/data_version`；
     - `required_timeframes` 与配置 `timeframes` 对齐，并包含 `execution_timeframe`。
3. M4-C：`run_manifest` v1 契约补齐
   - 目标：
     - 保持 `manifest_version=1`；
     - 补齐 Viewer 所需最小索引字段（`run_status/exit_code/trial_summary/config_refs/artifacts`）。
   - 实施要点：
     - 新增 `viewer_contract` 区段，显式标记 `required_files` 与 `optional_files` 状态；
     - 必需文件缺失时将 `viewer_contract.status` 标记为 `INVALID_RUN`；
     - 可选文件缺失时标记 `NOT_AVAILABLE`，不影响主结果状态。
4. M4-D：必需/可选文件判定与降级策略
   - 目标：
     - 将“缺失必需文件=阻断展示、缺失可选文件=降级展示”编码化并可审计。
   - 实施要点：
     - 必需集合至少包含：
       - `run_manifest.json`
       - `artifacts/summary.parquet`
       - `artifacts/signals.parquet`
       - `artifacts/trades.parquet`
       - `artifacts/equity.parquet`
       - `data_snapshot/dataset_index.json`
       - `data_snapshot/snapshot_meta.json`
       - `data_snapshot/raw/{execution_timeframe}.parquet`
     - 可选集合至少包含：
       - `artifacts/diagnostics.parquet`
       - `feature_catalog.json`
       - `precompile_report.json`
5. M4-E：测试与回归门槛
   - 自动化测试最小集：
     - 正例：单 trial 成功运行后，标准化目录与 manifest 契约完整；
     - 反例：缺失必需 timeframe 快照时，`viewer_contract.status=INVALID_RUN`；
     - 降级：可选文件缺失时标记 `NOT_AVAILABLE`。
   - 回归要求：
     - 不破坏既有 M1~M3 单测；
     - 保持 `task_guard.py check XTR-019` 通过。
6. M4-F：M4 退出定义（DoD）
   - Validation 的 M4 必跑项（目录契约/manifest 索引/快照索引/降级规则）均有正反例证据；
   - `task_guard.py check XTR-019` 通过；
   - M3 既有回测闭环语义（`next_bar_open`、`SKIPPED` 统计）不回退。

### M5 详细实施计划（实施级）
1. M5-A：统一哈希内核与序列化规范
   - 目标：
     - 在 Runtime 与 Precompile 统一使用同一套 Canonical JSON + 浮点规范化哈希实现。
   - 实施要点：
     - 抽取共享 hash 工具模块；
     - `config_hash/catalog_hash/params_hash` 全部走 `sha256:<64hex>`；
     - 明确排除非语义字段（运行时间/路径/日志/耗时）。
2. M5-B：`code_version` 生成与校验
   - 目标：
     - 生成符合 `git:<full_commit_sha>[+dirty]` 的 `code_version`。
   - 实施要点：
     - 默认从仓库 `HEAD` 探测 commit；
     - 支持运行参数覆盖（用于离线/受限环境）；
     - 格式非法或无法生成时 fail-fast。
3. M5-C：`data_version` 语义哈希闭环
   - 目标：
     - 由 `dataset_index + snapshot_meta` 规范化内容生成 `data:sha256:<64hex>`。
   - 实施要点：
     - `data_version` 计算输入剔除非语义字段（如 `generated_at`、绝对路径）；
     - 写入 `snapshot_meta`、`run_manifest`、`artifacts/summary` 并保持一致。
4. M5-D：运行性能日志与稳定性基线
   - 目标：
     - 输出 `elapsed_ms/peak_rss_mb/trials_count/bars_count/status`；
     - 支持稳定性回归验证（同输入多次运行 hash 一致）。
   - 实施要点：
     - Runtime 记录 run 级性能字段并落盘到 `run_manifest`；
     - 增加稳定性测试：两次同输入运行 `config_hash/catalog_hash/data_version` 一致。
5. M5-E：非功能阈值脚手架
   - 目标：
     - 提供可复用性能验证入口，支撑后续 P2 阈值验收。
   - 实施要点：
     - 在 Validation 中固化性能命令模板与日志字段；
     - 保证当前 CI 至少能产出结构化性能证据（阈值判定可分批完善）。
6. M5-F：M5 退出定义（DoD）
   - Validation 的 M5 必跑项（hash/version 一致性 + 性能日志 + 稳定性）有正反例证据；
   - `task_guard.py check XTR-019` 通过；
   - 不破坏 M4 既有产物契约与 Viewer 状态判定语义。

### 执行节奏（v1）
- 不删减当前 Spec/Validation 条目，仅按里程碑分批执行，避免一次性全量落地导致风险聚集。
- 每个里程碑至少覆盖“1 条正例 + 1 条反例”，并在 Validation Execution Log 记录命令、输出与证据路径。
- 阶段未达退出门槛不得进入下一里程碑实现，保证契约先稳再扩展能力。

## Acceptance
1. 声明式配置可驱动一次完整回测运行，无需手工拼接脚本逻辑。
2. precompile 阶段可自动生成 `feature_catalog`，并对规则引用执行 fail-fast 校验。
3. 参数变化但 `instance_id` 不变时，规则无需改写仍可正确运行。
4. 删除或重命名被引用 `instance_id` 时，运行前即报错并阻断执行。
5. 每个 trial 产物独立且包含 `resolved_config`、`feature_catalog`、`precompile_report`。
6. run 结果可追溯至 `trial_id/config_hash/catalog_hash`。
7. 保持现有动作输出 schema 与 `event_driven` 回测接口兼容。
8. 至少提供 1 条单 trial 路径与 1 条多 trial 路径的自动化验证用例。
9. 多输出 `output_key` 违规（例如 `macd.foo`）时，precompile 必须报结构化错误并阻断执行。
10. 运行期规则读取必须经过 `FeatureAccessor`，不允许直接依赖物理列名。
11. 配置主格式必须为 JSON，并通过 `schema_version=xtr_runtime_v1` 校验。
12. `trial_config.mode=scenarios` 可正确覆盖基线配置并生成独立 trial 运行结果。
13. 缺省 `trial_config` 时系统按 baseline 单 trial 运行，且产物完整。
14. 缺省 `scoring_rules` 时系统按非评分型路径运行，不阻断执行。
15. 配置出现未知字段时，加载阶段必须 fail-fast 并返回结构化错误。
16. 若 `execution_timeframe` / `timeframes` / `indicator_plan_by_tf` 存在一致性冲突，加载阶段必须阻断执行。
17. precompile 失败输出必须包含结构化错误数组，且每条错误至少含 `code/stage/severity/path/message`。
18. 约束类错误必须命中约定错误码（例如缺字段=`PC-CFG-001`，非法 `output_key`=`PC-OUT-001`）。
19. 多周期特征对齐必须满足 `source_timestamp + source_timeframe <= t_exec`，禁止任何前视读取。
20. 同周期特征读取必须表现为 1-bar lag，不允许读取当前未收盘 bar 的特征值。
21. v1 不支持 `source_timeframe < execution_timeframe` 的逆向映射，配置阶段必须阻断并报错。
22. 对齐缺失值默认保留 `NaN`，不允许在 Runtime Core 隐式前向填充。
23. `FeatureRef` 不存在时，`FeatureAccessor` 必须立即抛错并中断运行。
24. `FeatureRef` 存在但值缺失时，`FeatureAccessor` 返回 `NaN`，由评分/信号层按“未满足/0分”统一处理。
25. Runtime Core 必须先完成“全部 trial precompile”，再进入执行调度。
26. 多 trial 运行顺序必须稳定且可复现（按 `scenarios` 声明顺序）。
27. 单 trial 失败策略为 `abort`，多 trial 失败策略为 `continue`，并输出失败 trial 清单。
28. 缓存 key 必须命中 `symbol + timeframe + data_range + family + params_hash + code_version + data_version` 全字段。
29. v1 不允许跨 run 复用缓存；run 结束后缓存必须清空。
30. run summary 必须输出 `succeeded_trial_ids/failed_trial_ids/skipped_trial_ids`。
31. 单次运行必须在 `runs/{run_id}/` 下产出完整数据文件，避免分散到独立 report 目录。
32. `strategy_config.raw.json` 与 `strategy_config.resolved.json` 必须同时落盘并可追溯。
33. Runtime Core 不生成独立 report 文档；Report Viewer 仅基于结构化数据文件完成展示。
34. 系统不得向 `report` 目录写入新的运行产物，运行产物目录以 `runs/{run_id}` 为唯一标准。
35. `run_manifest` 仅保留配置引用（`config_refs`），不得重复存储配置细节参数。
36. `runs/{run_id}` 必须包含 `data_snapshot/raw/{timeframe}.parquet` 与 `data_snapshot/resampled/{timeframe}.parquet`，并覆盖 `required_timeframes`。
37. `runs/{run_id}` 必须包含 `data_snapshot/dataset_index.json` 与 `data_snapshot/snapshot_meta.json`，用于索引周期数据集与标注数据来源/规则/版本。
38. Viewer 若缺失必需文件或缺失任一 `required_timeframe` 数据集，必须返回 `INVALID_RUN`；缺失可选文件返回 `NOT_AVAILABLE`。
39. `warn_policy` 默认值必须为 `record_only`，且可显式切换为 `error`。
40. `warn_policy=record_only` 时，`WARN` 不阻断运行但必须写入 `precompile_report` 与 `run_manifest`。
41. `warn_policy=error` 时，任一 `WARN` 必须升级为失败并阻断运行。
42. `run_manifest` 必须输出 `run_status/exit_code/trial_summary.total_trials`，并与 trial 执行结果一致。
43. `warn_policy=record_only` 下，仅出现 `WARN` 不得单独导致 `run_status=FAILED`。
44. `warn_policy=error` 下，`WARN` 可导致 `run_status=FAILED` 且 `exit_code=1`。
45. `config_hash/catalog_hash/params_hash` 必须使用 `SHA-256` + Canonical JSON 计算，并满足跨运行一致性。
46. 非语义运行时字段不得参与 `config_hash/catalog_hash/params_hash` 计算。
47. `code_version` 必须符合 `git:<full_commit_sha>[+dirty]` 格式，且来源于当前代码版本。
48. `data_version` 必须符合 `data:sha256:<64hex>` 格式，且由 `dataset_index + snapshot_meta` 规范化内容生成。
49. 缺失或无法生成 `code_version/data_version` 时，运行必须中止并返回结构化错误。
50. `run_manifest` 与 `summary` 必须可追溯到同一 `code_version/data_version`。
51. `summary/trades/equity` 必须满足约定最小列级 schema；列缺失或类型错误必须判定 `INVALID_RUN`。
52. 基准场景 A 必须满足性能阈值：总耗时 `<= 60s` 且峰值内存 `<= 2GB`。
53. 基准场景 B 必须满足性能阈值：总耗时 `<= 10min` 且峰值内存 `<= 6GB`。
54. 标准场景下单次 precompile 必须 `<= 5s`。
55. 相同输入重复运行 3 次时，`run_status` 与 `config_hash/catalog_hash` 必须一致。
56. 相对基线版本性能退化超过 `20%` 时判定为不通过。
57. 验证记录必须包含 `elapsed_ms/peak_rss_mb/trials_count/bars_count/status` 字段。
58. `diagnostics.parquet` 若存在，必须满足约定最小 schema 与类型约束，并覆盖最小诊断语义（`score/state/weight/signal/reason`）。
59. `diagnostics.parquet` 缺失时标记 `NOT_AVAILABLE`；存在但 schema 不匹配时仅 diagnostics 子面板报错，不影响主结果面板。
60. `stop_loss/take_profit` 在动作输出层必须统一为绝对价格位（float64）；`%/ATR` 仅允许作为策略内部推导方式，执行层不得重复换算。
61. `size` 在动作输出层必须统一为账户权益占比（`0~1`）；`ENTER_*` 必须 `size>0`，`EXIT/HOLD` 必须 `size=0`，方向由 `action` 唯一决定。
62. `action` 在 v1 仅允许 `ENTER_LONG/ENTER_SHORT/EXIT/HOLD`；不允许 `REVERSE`，且同 bar 不得出现“平仓+开仓”双动作。
63. `reason_code` 在 v1 必须命中固定枚举，且与 `action` 满足 `ENTRY_*/EXIT_*/HOLD_*` 一致性约束；`reason_detail` 仅作解释不得驱动执行逻辑。
64. `risk_rules` 在 v1 必须满足固定 schema：`position_size.mode=fixed_fraction(value∈(0,1])`、`stop_loss.mode=atr_multiple(n>=1,k>0)`、`take_profit.mode=rr_multiple(rr>0)`；非法配置必须 fail-fast 并命中 `PC-CFG-003`。
65. v1 的 scenario `changes` 仅允许覆盖指标参数（`timeframe+instance_id+params`），不得覆盖 `risk_rules`；违规必须 fail-fast 并命中 `PC-TRI-001`。
66. `run_manifest` 必须包含整数 `manifest_version>=1`；v1 解析与校验基于 `manifest_version=1` 的显式分支执行。
67. 对 `manifest_version=1`，v1 最小必填字段允许新增但不得删除/重命名/变更语义；缺失任一必填字段必须判定 `INVALID_RUN`。
68. `run_manifest` 字段弃用必须遵循“两阶段”流程：先 deprecated 并保持读取，再于下一大版本移除。
69. `diagnostics.score_breakdown/weights_applied` 在 v1 必须为 JSON 对象字符串；值为数值，且 `weights_applied` 权重和满足 `abs(sum-1)<=1e-6`。
70. `dataset_index.datasets[].kind` 在 v1 仅允许 `raw/resampled`；`label_semantics` 仅允许 `open_time/right_label`；`timezone` 固定 `UTC`；`path` 必须为 run 内相对路径。
71. `snapshot_meta.required_timeframes` 在 v1 必须是 `timeframes` 子集，且必须包含 `execution_timeframe`；不满足时必须判定 `INVALID_RUN`。
72. `warn_policy` 在 v1 仅允许 `record_only/error`，且生效优先级固定为 `CLI > config > default`；同一 run 不支持 per-trial 覆盖。
73. `trial_config.scenarios` 在 v1 必须满足：`trial_id` 唯一、单 scenario 内 `timeframe+instance_id` 不重复；冲突必须 fail-fast 并命中 `PC-TRI-002`。
74. `trial_config.scenarios[].changes` 在 v1 必须原子生效：任一 change 非法时整个 scenario 失败，不得部分应用。
75. 执行时序在 v1 必须满足：`signal_time=t`、`execution_time=t+1`，成交价采用 `next_bar_open`，不允许同 bar 成交。
76. `stop_loss/take_profit` 在 v1 仅从入场后的后续 bar 生效，不得在入场同 bar 触发。
77. 数据尾部缺失 `t+1` bar 时，信号必须标记为未成交（skipped）并记录原因，不得伪造成交。
78. `artifacts/signals.parquet` 必须落盘，且至少包含 `status(FILLED|SKIPPED)` 与 `skip_reason`（当 `status=SKIPPED` 时）。
79. `artifacts/trades.parquet` 仅允许包含已成交记录，不得混入 `SKIPPED` 信号。
80. `artifacts/summary.parquet` 必须包含 `skipped_signal_count`，且与 `signals.status=SKIPPED` 计数一致。
81. 仅出现 `SKIPPED` 信号不得单独导致 `run_status=FAILED`。
82. `signals.skip_reason` 在 v1 必须命中固定枚举（`NO_NEXT_BAR|MARKET_CLOSED|DATA_GAP|RISK_BLOCKED|INVALID_ORDER`），且满足 `SKIPPED=必填`、`FILLED=为空`。
83. `RuntimeCore.run` 与 `PrecompileEngine.compile` 在 v1 必须满足冻结签名约束（含 `compile(config, trial_selector=None)`）。
84. `RuntimeRunResult` 与 `PrecompileResult` 在 v1 成功/失败路径均必须返回最小字段集（缺失视为契约不满足）。
85. v1 仅允许在接口返回中做向后兼容新增；接口/字段破坏性变更必须升级 `schema_version`。
