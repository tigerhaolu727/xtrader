# BLG-0002: timestamp 重采样右标签语义一致性

## Context
- Source discussion:
  - 围绕 `XTR-018`（指标层数据契约）讨论 `timestamp` 语义时发现：
  - 底层 K 线存储以 `open_time_ms` 为主（开盘时刻），但部分重采样/脚本路径存在“右标签（right label）”语义。
- Problem or opportunity:
  - 同为 `timestamp` 字段，可能在不同路径分别代表“bar open”或“bar right edge/close edge”，语义不一致。
  - 这会增加特征拼接、回测对齐、UI 展示与跨模块解释成本，也可能埋下时间错位风险。
- Why not now:
  - 当前优先目标是完成 `XTR-018` 指标层实现；评分系统与更完整时序契约尚未进入开发阶段。

## Desired Outcome
- 形成统一且可验证的时间语义契约：
  - 明确 `timestamp` 是否全链路强制为 `bar open time`（`datetime64[ns, UTC]`）。
  - 若历史路径存在右标签，提供兼容与迁移策略，避免破坏既有回测产物与 UI 读取。

## Proposed Direction
- Option A:
  - 统一输出为 `bar open` 语义，重采样阶段改为与开盘标签对齐。
  - 必要时额外保留 `bar_close_time` 字段用于可解释性。
- Option B:
  - 保留现有右标签路径，但在产物中显式声明 `timestamp_semantics` 元数据，并在消费端做归一化。
- Recommended:
  - 优先 Option A（统一语义），并采用“兼容期 + 校验告警”平滑迁移。

## Trigger Condition
- Start implementation when:
  - 评分系统开发启动，需要严格依赖时间对齐；
  - 或出现任一时序错位问题（特征/信号/执行/UI 对不齐）；
  - 或准备引入多周期联动策略并需要统一时间契约。
