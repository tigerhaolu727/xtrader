## Context
- Source discussion:
  - 围绕“评分系统与策略接口如何结合”讨论：是否必须引入 `StrategyProfile`，以及评分规则应放在何处执行。
  - 当前结论倾向：先不引入 `StrategyProfile`，采用策略内嵌评分引擎调用模式。
- Problem or opportunity:
  - 若每个策略都自行拼接评分逻辑，容易重复开发且接口不一致。
  - 若过早引入大而全抽象（如完整 profile 体系），会提高落地门槛并拖慢策略研发。
- Why not now:
  - 仍处于 `trading_strategy` 整体分析阶段，先沉淀最小可落地接口方案，暂不编码。

## Desired Outcome
- 冻结评分型策略 v1 接口方案，满足：
  - 不改现有 `BaseActionStrategy` 主接口；
  - 策略可在 `generate_actions` 内调用统一评分模块；
  - 支持“使用评分系统”与“不使用评分系统”两种策略并存；
  - 输出动作表契约保持统一（兼容现有回测链路）。

## Proposed Direction
- 接口层：
  - 保持 `BaseActionStrategy` 不变（`spec()` + `generate_actions(context)`）。
  - 评分型策略在类构造或配置中绑定 `scoring_rule_id`（或规则对象）。
- 执行层：
  - 在 `generate_actions` 内完成：
    1. 读取 `features` 输入；
    2. 调用评分模块（含市场状态判定与权重计算）；
    3. 输出 `signal/strength/reason`；
    4. 映射为标准动作列（`action/size/stop_loss/take_profit/reason`）。
- 参数与配置：
  - `params` 保留数值阈值与开关；
  - 复杂评分规则通过 `rule_id` 引用配置注册表，不直接塞入 `params`。
- 可解释性：
  - 评分细节先放 `diagnostics`（后续可升级为独立 run 产物表）。

## Trigger Condition
- Start implementation when:
  - 新策略进入编码阶段且决定采用评分体系；
  - 或需要在同一框架下同时支持“评分型策略”和“直判型策略”；
  - 或回测/UI 需要标准化输出评分解释字段。
